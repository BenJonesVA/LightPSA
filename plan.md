# PSA / Ticketing System — Build Plan

Status as of 2026-07-15. This tracks what's built, what's left, and the decisions behind it so future work (by you, me, or an agent) doesn't have to rediscover them.

## Stack

- Next.js 15 (App Router) + TypeScript + Tailwind CSS v3
- PostgreSQL + Prisma 6 (`^6.19.3`, deliberately pinned below the v7 beta — v7 moved datasource config to `prisma.config.ts` and would break the current setup)
- NextAuth v5 beta (`next-auth@5.0.0-beta.31`) + bcryptjs, JWT sessions, no adapter (no `Account`/`Session`/`VerificationToken` tables — not needed without OAuth/magic-link)
- Local dev: `docker-compose.yml` (Postgres 16 + Adminer)

## Done: Phase 1 — Database & Core Ticketing (MVP)

- Full schema in `prisma/schema.prisma`: `User`, `Client`/`Contact` (with parent/child company hierarchy), `Board`/`BoardMember`, `Category` (self-relation taxonomy), `CannedResponse`, `Ticket`, `TicketComment`, `TimeLog`, `Expense`, `Contract`/`ContractRate`.
- Postgres best-practices applied: `@db.Timestamptz(3)` on every timestamp (not naive `timestamp`), FK indexes, composite indexes on hot query paths (`Ticket(boardId,status)`, `Ticket(status,priority)`). Kept `cuid()` PKs over sequential bigints deliberately — contacts are client-portal-exposed, enumeration resistance matters more than the marginal index-locality win at this scale. `Ticket.id` is a sequential `Int` (customer-facing as `TKT-####`).
- CRUD UI: dashboard (`app/page.tsx`), Boards (`app/boards/*`), Tickets (`app/tickets/*` — list/detail/create/comments/status change), Clients & Contacts (`app/clients/*`).
- `prisma/seed.ts` — realistic MSP sample data (4 staff, 4 boards, 9 categories, 4 clients incl. one parent/child pair, 9 contacts, 3 contracts covering all three contract types, 13 tickets, comments, time logs, expenses, canned responses).
- `docker-compose.yml` for local Postgres + Adminer (port 5432/8080).

## Done: Phase 2 (partial) — Auth & RBAC

- NextAuth v5 with the Edge-safe config split: `auth.config.ts` (Edge-safe, imported by `middleware.ts`) vs `auth.ts` (Node-only, adds the bcrypt/Prisma `Credentials` providers). Two providers: `staff-login` (against `User`) and `client-login` (against `Contact`, requires `portalAccess: true`).
- `middleware.ts` — coarse gate only: unauthenticated → `/login`; wrong actor type for the area (`/portal/*` vs everything else) → bounced to the correct home. **Not** a substitute for in-action checks.
- `lib/rbac.ts` — `requireStaff()`, `requireRole(...roles)`, `requireClientSession()`. Every Server Action and role-sensitive page calls these directly; this is the actual authorization boundary. Boards/Clients creation is ADMIN/MANAGER-only; Tickets are open to all staff roles.
- `passwordHash` added to `User` and `Contact` (nullable — a contact without one, or with `portalAccess: false`, cannot authenticate). Seed data hashes a shared dev password (see below).
- `app/login`, `app/unauthorized`, session-aware `components/nav-shell.tsx`, `components/logout-button.tsx`.
- Minimal `app/portal/page.tsx` — read-only landing (contact's own open tickets) that proves the client-actor login path end-to-end. Deliberately not the full portal feature set (see below).
- Verified with real login flows (positive + negative) against a throwaway Postgres container each time changes landed — not just "pages render," but actual denials: wrong password, wrong role, wrong actor type, `portalAccess: false`.

### Fixed: stale JWT sessions crashed writes instead of failing gracefully

Found via a docker-logs review — `psa-postgres-1`'s log had 4 real, recent errors: `insert or update on table "TicketComment"/"TimeLog" violates foreign key constraint ..._fkey`, `Key (authorUserId/userId)=(...) is not present in table "User"`. This was a genuinely active bug, almost certainly hit by a real browser session (yours) still open from before this dev DB's Postgres volume was recreated earlier in the project.

- **Root cause**: JWT sessions with no DB adapter mean `session.user.id` (from `token.sub`) is stamped once at sign-in and never re-checked. If the underlying `User`/`Contact` row disappears (DB reset/reseed) or gets deactivated after that, the token's signature is still valid — Auth.js has no way to know the row is gone — so `requireStaff()`/`requireClientSession()` kept treating it as a good session. The first write that trusted `session.user.id` as a foreign key (adding a comment, logging time) then hit a raw Prisma constraint violation and 500'd instead of bouncing the user to `/login`.
- **Fix, in `auth.ts` only** (not `auth.config.ts`, which is Edge-safe and shared with `middleware.ts` — no Prisma access there): overrode the `jwt` callback to re-check, on every request that isn't a fresh sign-in, that the underlying `User` (`isActive`) or `Contact` (`isActive` + `portalAccess`) row still exists, returning `null` to end the session cleanly if not. `middleware.ts` still only does its original coarse check (unchanged, by design) — the real fix lives in the same place the rest of this app's authorization boundary already lives.
- **Second, smaller bug found while verifying the first**: `app/page.tsx` (the dashboard) was the *only* page in the entire app that called none of `requireStaff()`/`requireRole()`/`requireClientSession()` — confirmed by grepping every `page.tsx` in the tree. It rendered for anyone middleware let through, with no `isActive` re-check at all, which is exactly how my first test of the fix produced a false negative (the dashboard doesn't call `auth()` through `lib/rbac.ts`, so it never exercised the new check). Fixed by adding the same `requireStaff()` call every other staff page already has.
- **Verified for real, not just by reasoning about the code**: created a throwaway active `User`, logged in for a real signed session, confirmed it worked; flipped `isActive` to `false` *while keeping the same cookie* (the exact real-world scenario, not a simulation); confirmed a request with that same cookie now redirects to `/login` instead of rendering — first against `/tickets` (worked), then against `/` (didn't, initially — which is what surfaced the dashboard gap above; after adding `requireStaff()` there, retested and confirmed it also redirects). Repeated the same before/after test on the client-portal side (throwaway `Contact`, revoked `portalAccess` mid-session, confirmed `/portal` now redirects). Confirmed a genuinely fresh, valid session still reaches `/`, `/tickets`, `/kb`, `/schedule`, `/reports` normally (no regression). Deleted both throwaway accounts afterward. Rebuilt and restarted the containerized `app` image with the fix.
- **What this means for you right now**: if your own browser tab was the source of those log errors, your very next action in the app will redirect you to `/login` once — that's the fix working, not a new bug. Log back in and everything will work normally.

## Done: Phase 4 (job-runner slice) — background sweeps for idle time & SLA breaches

- **Job runner is a plain bearer-secret-protected route handler, not a Vercel-specific mechanism.** `lib/cron-auth.ts` checks `Authorization: Bearer $CRON_SECRET`; `vercel.json` wires it to Vercel Cron (every 15 min) if that's where this deploys, but *any* scheduler that can send an HTTP header works identically — a plain crontab + `curl`, GitHub Actions on a schedule, cron-job.org. This means the earlier "job scheduling blocks on picking a deployment target" framing was too strong — deployment target still matters for other reasons, but the job runner itself no longer depends on that choice.
- `GET /api/cron/idle-sweep` — drives the `IDLE_TIME_EXCEEDED` trigger, the one that was previously permanently disabled in `lib/automation.ts`. Exempted from the session-based `middleware.ts` gate (a scheduler has no browser session); the bearer check in the route itself *is* the authorization boundary for this surface, same role `requireRole()`/`requireStaff()` play for session-based Server Actions.
- `AutomationRule.conditionIdleMinutes` (new nullable column) — only meaningful for `IDLE_TIME_EXCEEDED` rules; required by both the create form and a server-side check in `createAutomationRule` when that trigger is chosen, ignored/nulled for every other trigger type. `IDLE_TIME_EXCEEDED` is now a selectable trigger in `/automation/new`.
- **Debounce is real, not assumed:** an idle rule that fires touches the ticket's `updatedAt` unconditionally after acting (`lib/automation.ts`), so it won't re-match and re-fire on every subsequent sweep — this matters most for `SEND_EMAIL_NOTIFICATION`, which doesn't touch the `Ticket` row on its own and would otherwise spam a new stub comment every 15 minutes forever. Verified directly: backdated a ticket, ran the sweep twice, confirmed exactly one stub comment existed after both runs, not two.
- `GET /api/cron/sla-breach-check` — proactive breach alerting on top of the read-time `getSlaStatus()` computation from the prior slice. Same no-email-provider-yet stub pattern: writes a marker-prefixed internal comment (`[SLA Breach: response]` / `[SLA Breach: resolution]`) instead of fabricating a real notification. Dedup is per breach kind via that marker string — verified by backdating a ticket past its resolution target, confirming both markers appeared on the first run, and confirming zero new alerts on a second run.
- **Verified functionally, not just rendered:** real HTTP calls against both endpoints — 401 with no/wrong bearer token, 200 with the correct one; backdated a real ticket's `updatedAt` past its rule's idle threshold and confirmed via direct DB query that the seeded "Escalate idle Emergency tickets to Marcus" rule reassigned it; also exercised the create-form's server-side validation directly (POST with `IDLE_TIME_EXCEEDED` and no `conditionIdleMinutes` → 500, no row created; same POST with a value → 303, row created correctly).

## Done: Phase 4 (email slice) — real outbound notifications via Resend

- `lib/email.ts` — thin wrapper around the Resend SDK. `sendEmail({to, subject, html})` never throws and never fabricates success: with no `RESEND_API_KEY`/`EMAIL_FROM` set it returns `{sent:false, reason:"..."}` immediately (safe default for local dev/CI/fresh checkouts); on a real send it returns `{sent:true}` only if Resend's API confirms it, otherwise the actual error message.
- `SEND_EMAIL_NOTIFICATION` (`lib/automation.ts`) now attempts a real send to the ticket's **client contact** (the natural recipient for a customer-ticket notification) when one exists. The internal audit-trail comment is still always written, but its wording now reflects the real outcome — sent successfully, no contact on file to notify, or attempted-but-failed with the actual reason — instead of a blanket "no provider configured" stub regardless of what actually happened.
- SLA breach alerts (`app/api/cron/sla-breach-check/route.ts`) now also email all active `ADMIN`/`MANAGER` staff (fetched once per sweep, not per ticket) in addition to the internal comment that remains the dedup source of truth. This is an ops alert, not a client-facing one — no contact involved.
- **Verified without a live Resend account** (no real API key available in this environment) by running the actual code path directly against a seeded ticket with `RESEND_API_KEY` unset: confirmed the exact honest failure message ("...but it did not go out: RESEND_API_KEY / EMAIL_FROM not configured") lands in the ticket's internal comment for a contact-linked ticket, and the correct "no client contact on file" message for a ticket with no contact — i.e. the three-way branch (sent / no recipient / send failed) is exercised and correct, just not against Resend's real API. **Whoever sets `RESEND_API_KEY`/`EMAIL_FROM` in a real environment should do one live smoke test** (trigger a status change on a contact-linked ticket, confirm the email actually lands) before relying on this for real client communication.

## Done: Phase 4 (close-out) — Inbound Email-to-Ticket via Postmark

Your call: Postmark's inbound webhook over the two alternatives (SendGrid inbound parse, IMAP polling) — real-time, no new mailbox-polling cron needed. Resend stays the outbound provider; Postmark is inbound-only, a second vendor by design since Resend doesn't do inbound parsing.

- **`app/api/inbound-email/route.ts`** — `POST`, exempted from `middleware.ts`'s session gate (`api/inbound-email` added to the matcher negative-lookahead alongside `api/cron` and `csat`) since Postmark has no browser session. Authorization is a `?token=` query param checked against `INBOUND_EMAIL_SECRET`, the same "secret in the URL is the boundary" pattern as `CRON_SECRET`'s bearer header and CSAT's unguessable id — you configure Postmark's webhook URL as `https://yourapp.com/api/inbound-email?token=<secret>`. Missing/wrong token → 401 (a real failure, surfaced immediately rather than left open by default). An unrecognized-sender or bad-ticket-reference outcome is a **200** with `{handled:false, reason:...}` in the body plus a `console.error` — those are application-level outcomes, not delivery failures, and returning non-2xx would just make Postmark retry the same unfixable email forever.
- **`lib/inbound-email.ts`** — `processInboundEmail(payload)`, the actual decision logic, kept separate from the route so it's testable/callable without an HTTP layer (same split as `lib/csat.ts`/`lib/automation.ts` vs. their route/action callers):
  - Sender is resolved via `Contact.email` (unique) — no match or `isActive: false` → rejected, logged, nothing written (there's no ticket yet to attach an honest-failure comment to, unlike the outbound email slice which always has one).
  - **Reply threading**: if any `ToFull` address matches `ticket-{id}@...`, the mail is appended as a `TicketComment` (`authorContactId`, `isInternal: false`) on that ticket instead of creating a new one. **Security-scoped, not identity-verified**: since `From:` headers are trivially spoofable, matching the reply address alone isn't treated as authorization — the sender's own `Contact.clientId` must equal the target ticket's `clientId`, which stops a spoofed header from letting one client's mail post onto a *different* client's ticket. It does **not** stop someone forging mail as an existing contact *of the correct client* — that's a real, documented limitation shared by most inbound-email parsers, not something this app's schema/design can close on its own.
  - **No reply match** → files a new ticket: `source: EMAIL` (an enum value that existed in the schema since Phase 1 but was unused until now), sender's own `clientId`/`contactId`, board defaults to "Support" (same fallback resolution as `createPortalTicket`), title from `Subject`, body from Postmark's `StrippedTextReply` (quoted-history-free) falling back to `TextBody`.
- **Reply-To wired on outbound client emails** — `lib/email.ts`'s `sendEmail()` now takes an optional `replyTo`; `lib/inbound-email.ts`'s `ticketReplyToAddress(ticketId)` builds `ticket-{id}@$INBOUND_EMAIL_DOMAIN` (returns `undefined`, not a broken address, until that env var is actually set). Wired into `SEND_EMAIL_NOTIFICATION` (`lib/automation.ts`) — the "there's an update on your ticket" email — so hitting Reply in a mail client threads straight back onto the ticket. **Deliberately not wired into the CSAT survey email** (`lib/csat.ts`) — that email's ask is "click this rating link," not "let's keep talking," and CSAT only fires once a ticket is already closed.
- **Two new env vars** (`.env.example`): `INBOUND_EMAIL_SECRET` (the webhook auth token) and `INBOUND_EMAIL_DOMAIN` (the domain Postmark's inbound server is configured on — must match what `ticket-{id}@` addresses resolve to).
- **Verified for real against the live containerized app + real seeded contacts, not simulated**: `tsc --noEmit` clean; rebuilt the `app` container with `INBOUND_EMAIL_SECRET`/`INBOUND_EMAIL_DOMAIN` set. Real HTTP POSTs to the running route: wrong token → 401; unrecognized sender (`nobody@unknown-domain.example`) → `handled:false`, nothing written; a real Acme contact (`diane.cross@acmecorp.com`) with no reply-address match → created ticket `TKT-17` with `source: EMAIL`, confirmed via a second real HTTP request (as ADMIN) that `/tickets/17` renders the real title/body/`Source: EMAIL`; the same contact replying to `ticket-17@...` → `handled:true, action:"commented"`, confirmed via direct DB query that a `TicketComment` landed with her `Contact` id as author; a BrightPath contact (`lauren.hayes@brightpathdental.com`) attempting the identical reply onto ticket 17 → rejected (`handled:false`, cross-client mismatch), confirmed via DB query that no second comment was written — the spoofing defense actually blocks the cross-client case it's designed for. Cleaned up the throwaway `TKT-17` afterward.

## Done: Phase 5 (partial) — CSAT survey

Triggered directly from the ticket-close code path (`app/tickets/actions.ts::updateTicketStatus`) rather than as a new admin-configurable `AutomationRule` action type — CSAT-on-close is a fixed product behavior, not something that needs board/priority/client conditions layered on it, so it reuses the automation engine's *infrastructure* (email send + honest-outcome internal comment, same pattern as `SEND_EMAIL_NOTIFICATION`) without becoming a new IFTTT rule kind.

- **Real bug found and fixed first, because CSAT depended on it**: `updateTicketStatus` never actually set `Ticket.resolvedAt`/`closedAt` — only `prisma/seed.ts` did. Every consumer that reads those fields (`lib/sla.ts`, the dashboard breach count, the new `/reports` page, and now CSAT's "just closed" trigger) was silently relying on a field the real app never wrote. Fixed: `updateTicketStatus` now stamps `resolvedAt` on the transition into RESOLVED/CLOSED and `closedAt` on the transition into CLOSED, and clears both on reopen — using `undefined` (not `null`) for "unchanged" so a redundant same-status submit or a later RESOLVED → CLOSED move doesn't reset an already-correct timestamp.
- **`CsatResponse` model** — one row per ticket (`ticketId` unique), `rating`/`comment` nullable until answered, `sentAt`/`respondedAt` tracked separately so "sent but not yet answered" and "never sent" render differently. `id` doubles as the public survey link token (`/csat/{id}`) — no separate token column, same reasoning as every other cuid PK already exposed externally in this schema (`Contact`, portal ticket ids).
- **`lib/csat.ts`** — `triggerCsatSurvey(ticket)`, called only on the OPEN-state → CLOSED transition. Idempotent by construction (`findUnique` guard before creating a row) so a reopen/close/reopen/close cycle sends exactly one survey, not one per close — the call site can't get this wrong even if it tried.

## Done: Containerizing the app

Previously only Postgres + Adminer ran in Docker; the Next.js app itself only ran via `npm run dev` on the host. Added a production-style container for the app, per explicit request — day-to-day coding still uses `npm run dev` (fast, hot reload); this is for prod-parity testing and eventual deployment, not for editing code inside a container.

- **`Dockerfile`** — 3-stage build (`deps` → `builder` → `runner`), using `next.config.mjs`'s new `output: "standalone"` so the runtime image only carries the node_modules actually traced from the server bundle, not the full tree. `openssl` is installed explicitly (`apk add`) — Prisma's query engine is linked against it and alpine doesn't ship it by default, a common Prisma-on-alpine gotcha. Runs as an unprivileged `nextjs` user, not root.
- **Migrations kept out of the runtime image on purpose.** The lean `runner` stage doesn't carry the Prisma CLI or the migrations folder — only `@prisma/client`'s generated engine, which the standalone build already traced in. Instead, `docker-compose.yml` has a `migrate` service that reuses the `builder` stage (which still has the full CLI + `prisma/migrations`), runs `prisma migrate deploy` once, and exits; `app` uses `depends_on: migrate: condition: service_completed_successfully` so `docker compose up` naturally sequences postgres → healthy → migrate → app, with no manual steps. This is fine for this single-instance setup; a multi-replica deployment would want migrations run as a distinct release step instead, to avoid every replica racing to migrate at once.
- **Seeding stays manual and out of the boot sequence** — `prisma/seed.ts` is dev sample data, not something that should run automatically every time a production-style container starts. Run it explicitly (`docker compose run --rm migrate npx prisma db seed`) the same way it's always been run.
- **DB connection inside the compose network** uses the `postgres` service name (`postgresql://psa:psa@postgres:5432/psa`), overriding the host-relative `DATABASE_URL` in `.env` (which points at `localhost:5432`, correct for the host dev server, wrong inside the Docker network) — set explicitly in `docker-compose.yml`'s `environment:` for both `migrate` and `app`, which takes precedence over the `env_file` values it also loads.

## Done: Deployment (self-hosted VM)

Your call: a self-hosted VM over a managed PaaS/full cloud. The existing production Docker image (see "Containerizing the app" above) needed a real story for TLS and public exposure to actually run on a bare VM — this slice is that story, split into three compose files layered on top of each other rather than one file that tries to be both dev and prod at once:

- **`docker-compose.yml` (base)** — no longer publishes any host ports itself (previously published Postgres 5432 and the app's 3131 directly, which would otherwise mean anyone hitting the VM's IP on those ports reaches Postgres/the app directly, bypassing TLS entirely). Postgres credentials are now `${POSTGRES_USER:-psa}` / `${POSTGRES_PASSWORD:-psa}` / `${POSTGRES_DB:-psa}` (Compose's own `.env`-interpolation, not `env_file:`) instead of hardcoded `psa`/`psa`, so a real deployment can set real credentials without editing the compose file.
- **`docker-compose.override.yml` (new, dev-only)** — republishes 5432/3131 and adds Adminer back. Compose auto-loads this file whenever you run `docker compose up` *without* explicit `-f` flags — which is exactly what the README's dev quick-start already does — so local day-to-day dev is unchanged: same command, same ports, same Adminer. The only thing that changed is *where* that dev-only exposure lives.
- **`docker-compose.prod.yml` (new, prod-only)** — adds Nginx Proxy Manager (`jc21/nginx-proxy-manager`) in front of `app`, published on 80/443 (plus 81 for its own admin UI). NPM was the explicit choice here over Caddy (tried first, reverted) — its GUI-driven proxy-host + Let's Encrypt cert setup is a better fit for a self-hosted VM one person manages by hand than a config file to hand-author. Used via `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`, which — critically — passing explicit `-f` flags like that is what stops Compose from *also* auto-loading `docker-compose.override.yml`; the dev-only file being present in the repo never leaks onto a real deploy as long as the deploy command always uses explicit `-f` flags (documented in the compose file's own header comment, not just here).
- **`AUTH_TRUST_HOST`** — new required env var for this path. Auth.js v5 rejects requests with an "UntrustedHost" error unless told to trust the `Host`/`X-Forwarded-*` headers a reverse proxy sets; Vercel sets this automatically, a self-hosted VM behind NPM has to set it explicitly.
- **`scripts/backup-db.sh` / `scripts/restore-db.sh`** — a nightly Postgres dump (gzip'd, 14-day retention by default) is *not* wired into docker-compose itself — backup cadence/retention/off-box copies are an operator decision, not something that should start/stop with the app stack. Meant to be driven by a host cron job (e.g. `0 3 * * * cd /opt/psa && ./scripts/backup-db.sh`), documented in the script's own header rather than duplicated here.
- **`scripts/test-prod-stack.sh`** — runs the *exact* same two compose files the real VM deploy uses, but under an isolated Compose project name (`psa-prod-test`) with its own containers/volumes/network, so it can be smoke-tested on a dev machine without ever touching the everyday dev stack. What it can't exercise is real TLS (Let's Encrypt needs a public domain with DNS actually pointed at a reachable IP, which `localhost` isn't) — that part of NPM's setup is inherently a one-time, browser-driven step on the real VM: visit `:81`, create the admin account (NPM 2.x has no default credentials — first boot reports `"setup": false` from its own health endpoint and requires creating the account through the UI, not an env var or API call), add a proxy host for your real domain → `app:3131`, request the cert.
