# PSA / Ticketing System — Build Plan

Status as of 2026-07-15. This tracks what's built, what's left, and the decisions behind it so future work (by you, me, or an agent) doesn't have to rediscover them.

## Stack

- Next.js 15 (App Router) + TypeScript + Tailwind CSS v3
- PostgreSQL + Prisma 6 (`^6.19.3`, deliberately pinned below the v7 beta — v7 moved datasource config to `prisma.config.ts` and would break the current setup)
- NextAuth v5 beta (`next-auth@5.0.0-beta.31`) + bcryptjs, JWT sessions, no adapter (no `Account`/`Session`/`VerificationToken` tables — not needed without OAuth/magic-link)
- Local dev: `docker-compose.yml` (Postgres 16 + Adminer)

## Done: Phase 1 — Database & Core Ticketing (MVP)

- Full schema in `prisma/schema.prisma`: `User`, `Client`/`Contact` (with parent/child company hierarchy), `Board`/`BoardMember`, `Category` (self-relation taxonomy), `CannedResponse`, `Ticket`, `TicketComment`, `TimeLog`, `Expense`, `Contract`/`ContractRate`.
- Postgres best-practices applied: `@db.Timestamptz(3)` on every timestamp (not naive `timestamp`), FK indexes, composite indexes on hot query paths (`Ticket(boardId,status)`, `Ticket(status,priority)`). Kept `cuid()` PKs over sequential bigints deliberately — contacts are client-portal-exposed, enumeration resistance matters more than the marginal index-locality win at this scale. `Ticket.id` is a sequential `Int` (customer-facing as `TKT-####`).
- CRUD UI: dashboard (`app/page.tsx`), Boards (`app/boards/*`), Tickets (`app/tickets/*` — list/detail/create/comments/status change), Clients & Contacts (`app/clients/*`).
- `prisma/seed.ts` — realistic MSP sample data (4 staff, 4 boards, 9 categories, 4 clients incl. one parent/child pair, 9 contacts, 3 contracts covering all three contract types, 13 tickets, comments, time logs, expenses, canned responses).
- `docker-compose.yml` for local Postgres + Adminer (port 5432/8080).

## Done: Phase 2 (partial) — Auth & RBAC

- NextAuth v5 with the Edge-safe config split: `auth.config.ts` (Edge-safe, imported by `middleware.ts`) vs `auth.ts` (Node-only, adds the bcrypt/Prisma `Credentials` providers). Two providers: `staff-login` (against `User`) and `client-login` (against `Contact`, requires `portalAccess: true`).
- `middleware.ts` — coarse gate only: unauthenticated → `/login`; wrong actor type for the area (`/portal/*` vs everything else) → bounced to the correct home. **Not** a substitute for in-action checks.
- `lib/rbac.ts` — `requireStaff()`, `requireRole(...roles)`, `requireClientSession()`. Every Server Action and role-sensitive page calls these directly; this is the actual authorization boundary. Boards/Clients creation is ADMIN/MANAGER-only; Tickets are open to all staff roles.
- `passwordHash` added to `User` and `Contact` (nullable — a contact without one, or with `portalAccess: false`, cannot authenticate). Seed data hashes a shared dev password (see below).
- `app/login`, `app/unauthorized`, session-aware `components/nav-shell.tsx`, `components/logout-button.tsx`.
- Minimal `app/portal/page.tsx` — read-only landing (contact's own open tickets) that proves the client-actor login path end-to-end. Deliberately not the full portal feature set (see below).
- Verified with real login flows (positive + negative) against a throwaway Postgres container each time changes landed — not just "pages render," but actual denials: wrong password, wrong role, wrong actor type, `portalAccess: false`.

## Done: Phase 2 — Client Portal (ticket slice)

- `app/portal/tickets/page.tsx` — full historical ticket list, scoped to the contact's `clientId`, optional `?status=` filter.
- `app/portal/tickets/[id]/page.tsx` — detail view. Ownership check and comment visibility are enforced **in the query itself**, not after fetching: `prisma.ticket.findFirst({ where: { id, clientId: user.clientId }, include: { comments: { where: { isInternal: false } } } })`. A wrong `clientId` → `notFound()`, never a fetched-then-hidden row.
- `app/portal/tickets/new/page.tsx` + `app/portal/actions.ts` (`createPortalTicket`, `addPortalComment`) — `clientId`/`contactId`/`source: PORTAL` are always set server-side from the session, never from form data; board is auto-assigned server-side (defaults to the "Support" board) so a contact can't file onto an internal-only board.
- **Decision made, not just deferred:** company hierarchy (`Client.parentId`) is **not** honored here — a parent-company contact does NOT see a child location's tickets. Scoping is strict `clientId` equality. "Parent sees children's tickets" remains a real, unmade product decision — if you want it, it needs to be added deliberately (e.g. resolve to a set of client IDs including descendants), not assumed.
- Verified with real logins against seed data: a comment marked `isInternal: true` on TKT-1 does not appear anywhere in the portal response (grepped the raw HTML, not just checked the rendered view — a CSS-hidden leak would still have shown up in that grep and didn't); a BrightPath ticket (TKT-4) 404s for an Acme contact; an Acme-Downtown-child ticket (TKT-2) also 404s for the Acme-parent contact, confirming the strict-scoping decision above actually shipped.
- Still open from the original Phase 2 list: attachments (needs a storage decision), password reset/invite flow (no self-service signup yet — fine for internal MVP, revisit before onboarding a real client).

## Done: Phase 3 (partial) — Time-Keeping, Contracts & Billing

- Time entry (`logTime`) and expense entry (`logExpense`) forms added to `app/tickets/[id]/page.tsx` / `app/tickets/actions.ts`. Both actions gate with `requireStaff()` and force `userId` to the logged-in technician's session id — there is no user/technician picker, a tech can only log time as themselves. `isApproved`/`isLocked` are never exposed to the submitter; they default to `false` and can only be flipped from the billing queue below.
- Contract consumption tracking on `app/clients/[id]/page.tsx` — for `RETAINER` contracts only, sums billable `TimeLog.durationMinutes` against `blockHoursPerPeriod` and renders a usage bar (red past 100%). **Explicitly labeled as an all-time total** ("per-period reset not yet implemented") rather than implying month-accuracy it doesn't have — there's still no billing-period-boundary table, so this is honest but not yet period-aware. `MSP_FLAT_RATE`/`TIME_AND_MATERIALS` contracts show no bar (nothing to consume against in this schema).
- Billing review queue at `/billing` (`app/billing/page.tsx` + `actions.ts`) — lists unlocked `TimeLog`/`Expense` rows across all clients with Approve/Lock actions. Gated `requireRole(ADMIN, MANAGER)` both at the page and as the first line of every action (`approveTimeLog`, `lockTimeLog`, `approveExpense`, `lockExpense`) — verified a TECHNICIAN session gets redirected to `/unauthorized` on `GET /billing`.
- Verified against real seeded data (mileage/material expense amounts, ticket titles) rendering correctly in the queue, and a real portal-created ticket + a real logged-time entry landing in the DB with server-derived (not client-supplied) ownership fields.

## Done: Phase 3 (close-out) — Billing periods & invoice export

- `lib/billing-period.ts` — `getCurrentBillingPeriod(contractStartDate, now?)` computes the current RETAINER billing period purely from the contract's `startDate` anchor day (no new schema/table). Handles the month-length edge case (a contract starting on the 31st correctly runs to the last day of short months, not day 3 of the next month).
- Contract consumption on `app/clients/[id]/page.tsx` is now period-scoped (`TimeLog.startTime` between the computed period bounds), with the caption showing the real date range instead of "all-time total." Verified rendering `Jun 30 – Jul 31` for Acme's retainer.
- CSV invoice export at `GET /billing/export` (`app/billing/export/route.ts`) — streams all `isLocked: true` `TimeLog`/`Expense` rows as a dated CSV. `requireRole(ADMIN, MANAGER)` gated; verified 200 for ADMIN (correct header row, empty body since nothing's locked yet in fresh seed data) and 307→`/unauthorized` for TECHNICIAN.
- Still no persisted `InvoiceBatch`/`InvoiceLineItem` tables — this is CSV-export-only by deliberate choice, not an oversight. Revisit only once an accounting sync target (QuickBooks/Xero) is chosen; building the API-shaped tables before knowing which API they feed would be guessing.

## Done: Phase 4 (event-driven slice) — SLA Engine & Automation

- `lib/sla.ts` — `getSlaStatus(ticket, policy, now?)`, pure and computed at read time (no stored timer state). First response is defined as the earliest **client-visible** (`isInternal: false`) comment authored by staff — an internal note doesn't count as "responding to the client." `SlaPolicy` (one row per `TicketPriority`, seeded with sensible defaults) is manageable at `/admin/sla` (ADMIN/MANAGER only).
- `lib/automation.ts` — `runAutomationRules(ticket, triggerType, now?)`, the IFTTT execution engine. **Re-entrancy is prevented by construction, not by convention**: it applies actions via direct `prisma.ticket.update` calls only, never by calling back through `createTicket`/`updateTicketStatus`/itself — so there is no code path where an action's effect could re-trigger the engine. Verified this by reading the file after all 4 build agents finished to confirm no recursive call sites exist.
- `AutomationRule` CRUD at `/automation` (ADMIN/MANAGER only).
- `SEND_EMAIL_NOTIFICATION` started as a labeled stub here — no email provider was configured yet at this point in the build. See the Phase 4 email slice below for the real implementation.
- Priority escalation is now a real, usable feature (previously there was no way to change a ticket's priority after creation at all, which would have made the `PRIORITY_ESCALATED` trigger dead on arrival) — `updateTicketPriority` only fires the trigger on a strict priority increase; downgrades update silently.
- Canned-response substitution is live on the ticket reply form (`{{client_name}}`/`{{ticket_id}}` filled from real data via a small client component).
- **Verified functionally, not just rendered:** created a real EMERGENCY-priority ticket via an actual HTTP POST through the seeded "Auto-assign Emergency tickets to Alice" rule and confirmed in the database that `assigneeId` was set to Alice — the full trigger → match → direct-update pipeline, not a page-load check. Also verified ADMIN/TECHNICIAN role gating on `/admin/sla` and `/automation` the same way as every other admin surface this session.

## Done: Phase 4 (job-runner slice) — background sweeps for idle time & SLA breaches

- **Job runner is a plain bearer-secret-protected route handler, not a Vercel-specific mechanism.** `lib/cron-auth.ts` checks `Authorization: Bearer $CRON_SECRET`; `vercel.json` wires it to Vercel Cron (every 15 min) if that's where this deploys, but *any* scheduler that can send an HTTP header works identically — a plain crontab + `curl`, GitHub Actions on a schedule, cron-job.org. This means the earlier "job scheduling blocks on picking a deployment target" framing was too strong — deployment target still matters for other reasons, but the job runner itself no longer depends on that choice.
- `GET /api/cron/idle-sweep` — drives the `IDLE_TIME_EXCEEDED` trigger, the one that was previously permanently disabled in `lib/automation.ts`. Exempted from the session-based `middleware.ts` gate (a scheduler has no browser session); the bearer check in the route itself *is* the authorization boundary for this surface, same role `requireRole()`/`requireStaff()` play for session-based Server Actions.
- `AutomationRule.conditionIdleMinutes` (new nullable column) — only meaningful for `IDLE_TIME_EXCEEDED` rules; required by both the create form and a server-side check in `createAutomationRule` when that trigger is chosen, ignored/nulled for every other trigger type. `IDLE_TIME_EXCEEDED` is now a selectable trigger in `/automation/new`.
- **Debounce is real, not assumed:** an idle rule that fires touches the ticket's `updatedAt` unconditionally after acting (`lib/automation.ts`), so it won't re-match and re-fire on every subsequent sweep — this matters most for `SEND_EMAIL_NOTIFICATION`, which doesn't touch the `Ticket` row on its own and would otherwise spam a new stub comment every 15 minutes forever. Verified directly: backdated a ticket, ran the sweep twice, confirmed exactly one stub comment existed after both runs, not two.
- `GET /api/cron/sla-breach-check` — proactive breach alerting on top of the read-time `getSlaStatus()` computation from the prior slice. Same no-email-provider-yet stub pattern: writes a marker-prefixed internal comment (`[SLA Breach: response]` / `[SLA Breach: resolution]`) instead of fabricating a real notification. Dedup is per breach kind via that marker string — verified by backdating a ticket past its resolution target, confirming both markers appeared on the first run, and confirming zero new alerts on a second run.
- **Verified functionally, not just rendered:** real HTTP calls against both endpoints — 401 with no/wrong bearer token, 200 with the correct one; backdated a real ticket's `updatedAt` past its rule's idle threshold and confirmed via direct DB query that the seeded "Escalate idle Emergency tickets to Marcus" rule reassigned it; also exercised the create-form's server-side validation directly (POST with `IDLE_TIME_EXCEEDED` and no `conditionIdleMinutes` → 500, no row created; same POST with a value → 303, row created correctly).

## Done: Phase 4 (email slice) — real outbound notifications via Resend

- `lib/email.ts` — thin wrapper around the Resend SDK. `sendEmail({to, subject, html})` never throws and never fabricates success: with no `RESEND_API_KEY`/`EMAIL_FROM` set it returns `{sent:false, reason:"..."}` immediately (safe default for local dev/CI/fresh checkouts); on a real send it returns `{sent:true}` only if Resend's API confirms it, otherwise the actual error message.
- `SEND_EMAIL_NOTIFICATION` (`lib/automation.ts`) now attempts a real send to the ticket's **client contact** (the natural recipient for a customer-ticket notification) when one exists. The internal audit-trail comment is still always written, but its wording now reflects the real outcome — sent successfully, no contact on file to notify, or attempted-but-failed with the actual reason — instead of a blanket "no provider configured" stub regardless of what actually happened.
- SLA breach alerts (`app/api/cron/sla-breach-check/route.ts`) now also email all active `ADMIN`/`MANAGER` staff (fetched once per sweep, not per ticket) in addition to the internal comment that remains the dedup source of truth. This is an ops alert, not a client-facing one — no contact involved.
- **Verified without a live Resend account** (no real API key available in this environment) by running the actual code path directly against a seeded ticket with `RESEND_API_KEY` unset: confirmed the exact honest failure message ("...but it did not go out: RESEND_API_KEY / EMAIL_FROM not configured") lands in the ticket's internal comment for a contact-linked ticket, and the correct "no client contact on file" message for a ticket with no contact — i.e. the three-way branch (sent / no recipient / send failed) is exercised and correct, just not against Resend's real API. **Whoever sets `RESEND_API_KEY`/`EMAIL_FROM` in a real environment should do one live smoke test** (trigger a status change on a contact-linked ticket, confirm the email actually lands) before relying on this for real client communication.

## Done: Phase 5 (partial) — Reporting/dashboards

`/reports` (ADMIN/MANAGER only, linked from the sidebar's Manage group). No new schema — pure aggregation over `Ticket`/`TimeLog`/`Contract`/`SlaPolicy`, which were already being collected by earlier phases.

- **KPI strip** (last 30 days): tickets created, tickets resolved, avg. resolution time, overall SLA compliance %.
- **Ticket volume trend** — weekly created-vs-resolved columns, last 8 weeks.
- **SLA compliance by priority** — reuses `lib/sla.ts`'s `getSlaStatus` (same source of truth as the dashboard's breach count and the ticket-detail SLA badge) rather than a separate compliance calculation, so this can't drift from what the rest of the app considers a breach.
- **Technician utilization** — billable vs. non-billable hours per tech, last 30 days.
- **Retainer consumption across all clients** — same per-contract billing-period logic as `app/clients/[id]/page.tsx`'s usage bar (via `getCurrentBillingPeriod`), just aggregated company-wide instead of one client at a time. Deliberately not extracted into a shared helper — it's a handful of lines and the two call sites aren't at risk of drifting apart silently, so a shared abstraction wasn't worth adding for this pass.
- **`components/ui/bar-chart.tsx` / `column-chart.tsx`** — new hand-rolled, dependency-free chart primitives (same philosophy as `badge.tsx`/`button.tsx`/`card.tsx`: no charting library, just divs + Tailwind tokens). `Bar` takes a `max` per call, not a shared scale, so it works for both "% of policy target" bars and "% of this contract's own block hours" bars without forcing an artificial common scale.
- **Bug caught and fixed during verification**: two seeded tickets have `resolvedAt` 1ms before `createdAt` (a seed-data timestamp artifact), which made the average-resolution-time KPI compute a tiny negative number and render as `−0m`. Fixed by clamping each ticket's resolution duration to `Math.max(0, ...)` before averaging — a ticket can't take negative time to resolve, so a near-instant resolution should read as `0m`, not a nonsensical negative value. (This is a distinct case from the SLA badge's use of `formatDuration`, where a negative value legitimately means "overdue" and the minus sign is the point.)
- **Verified for real:** `tsc --noEmit` clean; logged in as ADMIN and confirmed real DB-matching numbers (13 tickets created / 2 resolved in the seeded 30-day window, SLA compliance 92% = 12/13, per-priority breakdown summing correctly, retainer bar showing Acme's real contract); logged in as TECHNICIAN and confirmed `/reports` still redirects to `/unauthorized`; confirmed the new chart classes resolve to real compiled CSS rules, not just class names with nothing behind them. Technician utilization and retainer usage both show empty/zero in the current seed data — not a bug, the seeded `TimeLog` rows are dated mid-2025, outside both the 30-day report window and the current 2026-07 retainer billing period.

## Done: Phase 5 (partial) — CSAT survey

Triggered directly from the ticket-close code path (`app/tickets/actions.ts::updateTicketStatus`) rather than as a new admin-configurable `AutomationRule` action type — CSAT-on-close is a fixed product behavior, not something that needs board/priority/client conditions layered on it, so it reuses the automation engine's *infrastructure* (email send + honest-outcome internal comment, same pattern as `SEND_EMAIL_NOTIFICATION`) without becoming a new IFTTT rule kind.

- **Real bug found and fixed first, because CSAT depended on it**: `updateTicketStatus` never actually set `Ticket.resolvedAt`/`closedAt` — only `prisma/seed.ts` did. Every consumer that reads those fields (`lib/sla.ts`, the dashboard breach count, the new `/reports` page, and now CSAT's "just closed" trigger) was silently relying on a field the real app never wrote. Fixed: `updateTicketStatus` now stamps `resolvedAt` on the transition into RESOLVED/CLOSED and `closedAt` on the transition into CLOSED, and clears both on reopen — using `undefined` (not `null`) for "unchanged" so a redundant same-status submit or a later RESOLVED → CLOSED move doesn't reset an already-correct timestamp.
- **`CsatResponse` model** — one row per ticket (`ticketId` unique), `rating`/`comment` nullable until answered, `sentAt`/`respondedAt` tracked separately so "sent but not yet answered" and "never sent" render differently. `id` doubles as the public survey link token (`/csat/{id}`) — no separate token column, same reasoning as every other cuid PK already exposed externally in this schema (`Contact`, portal ticket ids).
- **`lib/csat.ts`** — `triggerCsatSurvey(ticket)`, called only on the OPEN-state → CLOSED transition. Idempotent by construction (`findUnique` guard before creating a row) so a reopen/close/reopen/close cycle sends exactly one survey, not one per close — the call site can't get this wrong even if it tried.
- **`/csat/[id]`** (`app/csat/[id]/page.tsx` + `app/csat/actions.ts`) — public, unauthenticated survey page. No session check: the unguessable `CsatResponse` id in the URL *is* the authorization boundary, same role the `CRON_SECRET` bearer check plays for the cron routes. Exempted from `middleware.ts`'s session gate accordingly. 1–5 rating + optional comment, plain HTML form (no client JS required); resubmitting an already-answered survey is a no-op (`updateMany` guarded on `respondedAt: null`) rather than overwriting the recorded answer.
- **Surfaced in two places**: a "Customer satisfaction" card in the ticket detail right rail (rating + comment once answered; an honest "why not sent" message otherwise — distinguishes "no contact on file" from "send failed for another reason," since those are genuinely different states); an avg-rating / response-rate / surveys-sent block on `/reports`.
- **Verified for real, end to end** — attempting to forge Next.js's server-action wire protocol (`$ACTION_*` encrypted-closure form encoding) via raw curl to drive `changeStatus` from the outside turned out to be impractical to reverse-engineer reliably; verified instead by directly invoking the real shipped functions (`triggerCsatSurvey`, `submitCsatResponse`) end to end against a real seeded ticket: confirmed `resolvedAt`/`closedAt` both stamped correctly on close, a `CsatResponse` row created, the real client contact found, `sendEmail` failing gracefully (no `RESEND_API_KEY` in this environment) and writing an honest internal comment naming the real reason, then submitting a real rating (4/5 + comment) and confirming it round-trips through both the staff ticket-detail card and the public `/csat/{id}` page (which correctly shows the "already responded" thank-you state, not the form, on a second view). Also confirmed: unknown survey id → 404 (no data leak), no forced login on a valid survey link, TECHNICIAN still gated from `/reports`, `tsc --noEmit` clean. One seeded `CsatResponse` (5/5) added for the one seeded CLOSED ticket so the feature has non-empty demo data out of the box.

## Done: Phase 5 (partial) — Knowledge Base

`KbArticle` model, following the exact `CannedResponse` pattern already in the schema (optional board association, `createdBy`, timestamps) plus an `isInternal` flag with the same semantics as `TicketComment.isInternal`.

- **Open to all staff, not just ADMIN/MANAGER** — creating and editing (`requireStaff()` only, no author-lock) is intentionally not restricted the way board/client creation is. A KB works best when frontline techs document the fixes they find; gating it to admins would discourage exactly that. Any staff member can edit any article, same trust model as ticket status updates.
- **Staff**: `/kb` (list, with board/category/text-search filters via `?boardId=&categoryId=&q=`, search is a plain case-insensitive `contains` — no full-text-search infra, not warranted at this scale), `/kb/new`, `/kb/[id]`, `/kb/[id]/edit`. No delete — matches this app's existing convention of never hard-deleting rows.
- **Portal** (`/portal/kb`, `/portal/kb/[id]`): `isInternal: false` is baked into the query itself, not filtered after fetching — same pattern as the portal ticket detail's internal-comment scoping, so an internal article can never leave the database in the response, let alone the rendered page. Not client-scoped — this is a shared help-center-style KB across all portal clients, since board/category are about ticket taxonomy, not client ownership, and nothing in this feature implies per-client content.
- Nav links added to both the staff sidebar (Workspace group) and the client portal topbar.
- **Verified for real:** `tsc --noEmit` clean; as ADMIN, confirmed `/kb` lists all 3 seeded articles including the internal one (correctly badged), the search filter (`?q=firewall`) returns only the matching article, and `/kb/[id]/edit` renders; confirmed KB is genuinely open to all staff by hitting `/kb` and `/kb/new` as TECHNICIAN (both 200, unlike the ADMIN/MANAGER-gated surfaces elsewhere in this app); as a real portal contact, confirmed the raw HTML of `/portal/kb` contains neither the internal article's title nor body, then requested the internal article's *real* id directly at `/portal/kb/{id}` and got a genuine 404 (not a redirect, not a hidden-but-fetched row) — the same "ownership check baked into the query" discipline as the rest of the portal; exercised the real `createArticle`/`updateArticle` write paths directly (create → update → delete) and confirmed each write landed correctly, then cleaned up the throwaway row; unauthenticated `/kb` redirects to `/login`. Rebuilt and restarted the containerized `app` image to pick up the new model, confirmed it came up clean against the same Postgres data.

## Done: Containerizing the app

Previously only Postgres + Adminer ran in Docker; the Next.js app itself only ran via `npm run dev` on the host. Added a production-style container for the app, per explicit request — day-to-day coding still uses `npm run dev` (fast, hot reload); this is for prod-parity testing and eventual deployment, not for editing code inside a container.

- **`Dockerfile`** — 3-stage build (`deps` → `builder` → `runner`), using `next.config.mjs`'s new `output: "standalone"` so the runtime image only carries the node_modules actually traced from the server bundle, not the full tree. `openssl` is installed explicitly (`apk add`) — Prisma's query engine is linked against it and alpine doesn't ship it by default, a common Prisma-on-alpine gotcha. Runs as an unprivileged `nextjs` user, not root.
- **Migrations kept out of the runtime image on purpose.** The lean `runner` stage doesn't carry the Prisma CLI or the migrations folder — only `@prisma/client`'s generated engine, which the standalone build already traced in. Instead, `docker-compose.yml` has a `migrate` service that reuses the `builder` stage (which still has the full CLI + `prisma/migrations`), runs `prisma migrate deploy` once, and exits; `app` uses `depends_on: migrate: condition: service_completed_successfully` so `docker compose up` naturally sequences postgres → healthy → migrate → app, with no manual steps. This is fine for this single-instance setup; a multi-replica deployment would want migrations run as a distinct release step instead, to avoid every replica racing to migrate at once.
- **Seeding stays manual and out of the boot sequence** — `prisma/seed.ts` is dev sample data, not something that should run automatically every time a production-style container starts. Run it explicitly (`docker compose run --rm migrate npx prisma db seed`) the same way it's always been run.
- **DB connection inside the compose network** uses the `postgres` service name (`postgresql://psa:psa@postgres:5432/psa`), overriding the host-relative `DATABASE_URL` in `.env` (which points at `localhost:5432`, correct for the host dev server, wrong inside the Docker network) — set explicitly in `docker-compose.yml`'s `environment:` for both `migrate` and `app`, which takes precedence over the `env_file` values it also loads.
- **Verified for real:** built and ran the actual image (`docker compose up -d`), confirmed the three-stage dependency chain executed as designed (postgres → healthy → migrate ran to completion and exited → app started), then logged in as ADMIN through the *containerized* app against the *containerized* Postgres and swept 10 staff pages (all 200) — including confirming `/tickets/1` and `/tickets/9` render the exact CSAT/close data written during the previous session's verification, proving this is the same persistent volume, not a fresh DB. Also confirmed TECHNICIAN still redirects to `/unauthorized` through the container, and Adminer is unaffected. Runtime `psa-app` image is ~341MB; the `psa-migrate` image (full toolchain, used only transiently) is ~1.36GB — expected, not a runtime cost.
